# 백엔드 DB 연동 계획서

## 목적
- `chatbot-backend`에 PostgreSQL 연동을 적용해 대화, 사용자, 세션 등 핵심 데이터를 영속화
- PoC 임시 인증 흐름(`dev-login`)을 유지하면서 정식 인증 전환에 대비
- 메인 채팅 흐름(`chatbot-front -> chatbot-backend -> flowise`)의 안정성/추적성을 강화

## 범위
- 백엔드: FastAPI + PostgreSQL
- 인증: `dev-login` 유지(환경 제어), 토큰 기반
- 데이터: 사용자/세션/대화/로그/설정 메타데이터
- 제외: `reference/*` 수정(참고 전용)

## 전제/제약
- Firebase 사용 금지
- Gemini 사용 금지
- `reference` 하위 프로젝트는 분석만 수행
- 로그인 방식은 `00.documents/analysis/auth-login-proposal.md` 준수

## 목표 산출물
- DB 스키마 설계서(ERD 포함)
- 마이그레이션 스크립트(초기 스키마)
- FastAPI DB 연동 모듈(연결/세션/리포지토리)
- 핵심 API의 DB 연동(유저/세션/대화/로그)
- 운영/개발 환경 분리 설정(.env, 연결 풀)

## 스키마 생성 원칙
- ORM 모델 정의 후 Alembic 자동 생성 사용
- 생성된 마이그레이션 SQL은 반드시 리뷰/수정
- 인덱스/제약은 마이그레이션에 명시적으로 작성

## 업무 단계

### 1) 요구사항 정리 및 데이터 모델 정의
- 사용자/세션/대화/메시지/로그의 엔터티 목록화
- 필수 필드, 인덱스, 보존기간 정책 정의
- `dev-login` 사용자 자동 생성 정책 확정

### 2) DB 스키마 설계
- 테이블/관계/키 설계(1차 정규화)
- 인덱스 전략(조회 성능/로그 적재)
- 삭제 정책(소프트 삭제 vs 하드 삭제) 결정

### 3) 연동 기술 스택 확정
- ORM/쿼리 계층: SQLAlchemy(권장) 또는 SQLModel
- 마이그레이션: Alembic
- 설정: `DATABASE_URL`, 풀 크기, 타임아웃

### 4) FastAPI 연동 구현
- DB 세션/의존성 주입 모듈 작성
- 리포지토리 계층(쿼리 캡슐화)
- 공통 트랜잭션 정책

### 5) 핵심 API DB 연동
- `dev-login` 유저 조회/자동 생성
- 대화 시작/종료, 메시지 저장
- 요청/응답 로깅(추적성 확보)

### 6) 데이터 보안/운영 고려
- PII 최소 수집, 마스킹 정책
- 감사 로그 및 접근 로그
- 환경별 DB 분리(DEV/QA/PROD)

### 7) 테스트 및 검증
- 마이그레이션 적용 테스트
- API 단위 테스트(로그인/대화 저장)
- 성능 기본 점검(대화 저장/조회)

## 데이터 모델 참고
- 상세 ERD와 테이블 정의는 `00.documents/analysis/erd.md`를 단일 기준으로 사용
- 본 문서는 연동 범위/단계 중심으로 유지

## API 연동 포인트(예시)
- `GET /auth/dev-login?id={user_id}`
  - 사용자 조회/생성 → 토큰 발급 → 사용자 기록 저장
- `POST /chat/start`
  - 세션/대화 생성
- `POST /chat/message`
  - 메시지 저장 → flowise 호출 → 응답 저장
- `POST /chat/end`
  - 세션 종료 처리

## 리스크 및 대응
- 대량 메시지 적재로 인한 성능 저하
  - 인덱스/파티셔닝 검토, 아카이빙 정책 수립
- 임시 로그인 보안 리스크
  - 환경 제어, IP 제한, 만료 토큰
- 스키마 변경 빈도 증가
  - 마이그레이션 규칙 및 릴리스 절차 확립

## 일정 초안(2~3주)
- 1주차: 요구사항/스키마/연동 구조 확정
- 2주차: DB 연동/핵심 API 적용
- 3주차: 테스트/성능 점검/문서화

## 결정 필요 사항
- 사용자 PK 타입(string vs uuid)
- 소프트 삭제 적용 범위
- 메시지 보관 기간 및 아카이빙 정책
- 환경별 DB 분리 정책
